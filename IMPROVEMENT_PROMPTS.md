# 每日改进建议生成 — 提示模板

> 用途：让 Claude Code 定时（你设定的时间）**只产出改进建议清单**，不直接改代码。
> 你早上看清单、挑选认可的，再让它实现。这样自动化的是"动脑暴"，落地由你把关。

---

## 给 Claude Code 的提示（建议模式 — 默认用这个）

```
先通读 CLAUDE.md 了解项目全貌。

然后扮演一位资深产品 + 工程顾问，针对「阅读探险队 Reading Quest」这个家庭阅读打卡应用，
独立思考并提出 1 个值得做的改进点。要求：

1. 从这些角度之一切入：孩子的使用体验、家长的需求、阅读教育价值、
   数据/报告的洞察力、留存与激励机制、可用性/无障碍、性能、代码健壮性。
2. 不要选 CLAUDE.md 里 TODO 已列出的（除非你有全新的具体做法）。
3. 给出：
   - 【改进点】一句话说清是什么
   - 【为什么】解决什么真实问题、对谁有价值
   - 【怎么做】具体实现思路（涉及哪些函数/数据结构）
   - 【风险】可能影响的现有功能、需要注意的兼容性
   - 【工作量】小/中/大
4. **不要修改任何代码**。只输出这份建议，写到 IMPROVEMENT_IDEAS.md 文件里
   （追加到顶部，带上今天日期）。

输出后停止，等我审阅。
```

---

## 给 Claude Code 的提示（实现模式 — 我挑好之后用）

```
先通读 CLAUDE.md。

我决定实现 IMPROVEMENT_IDEAS.md 里 [日期/编号] 这一条改进：[粘贴改进点]。

请按 CLAUDE.md 的规范实现：
1. 改 index.html，升 APP_VERSION。
2. 如果改了积分/题库/校验，同步更新家长设置说明卡。
3. **必须跑完整执行测试**（见 CLAUDE.md 测试规范）：真实执行 finishCheckin 全路径、
   所有 renderX、扫描所有 onclick 函数是否定义、用例间清空 DOM stub。全绿才算完成。
4. 测试全过后，把这次改动和踩到的坑追加写回 CLAUDE.md 的相应章节。
5. 提交并推送到 main。

实现完告诉我版本号，以及我该怎么验证。
```

---

## 为什么默认用"建议模式"而不是"直接改"

无人值守直接改生产代码风险高（行业共识：生成代码的 agent 不该自己给自己打分）。
这个项目已经复杂、带云端真实数据，一旦改坏影响全家使用。
"建议模式 + 你审阅 + 实现模式"把自动化用在最安全的环节（动脑暴），
落地始终有你这一道人工关。等你对它的产出质量有信心了，再考虑放宽。
